Overall Verdict
Solid foundation. Real gaps in the three pillars that matter.

SwimFitPro is a well-engineered single-file PWA with genuinely smart features — position-based periodization, adaptive loading from readiness/sleep, tempo notation, progressive 1RM tracking, and age-specific Canadian standards. That is more than most swimmer tools do. But the app was designed for dryland and the other two stated pillars — time standards progression and nutrition — are underbuilt relative to what elite swimmers actually need. The fuel tab is a calorie estimator, not a nutrition tool. The times tab has no in-progress season arc or target-date projection. And there are two active bugs (unit picker, km tracker unit bug) that undermine trust in the data. Fix the bugs first, then build out the gaps in order.

What's Built & Working Well

Strengths
Position-Based Periodization

The goalPhaseFromPosition() logic is genuinely good. It correctly scales any block length into GPP → Stimulus → Strength → Sharpen → Taper, rather than collapsing short blocks into a premature taper. Most apps don't do this at all — they just count down weeks and do nothing with the data.

Age-Specific Canadian Standards

Four-tier standards hierarchy (Prov Champ → Prov Trials → Canada Open → Canada Trials), Swimming Canada's Dec 31 competition age rule correctly implemented, Junior vs Senior Trials auto-selection based on comp age. The standards data is accurate and up to date.

Adaptive Load from Readiness

Sleep + readiness sliders feeding directly into set/volume adjustments is the right call. It mirrors what elite athletes do manually. The 7-day average tracking with a debounced save adds genuine longitudinal value.

1RM Progressive Loading

Phase-aware percentage targets (GPP 65% → Strength 85%), Epley formula for estimation, per-exercise history log, and automatic weight progression suggestion when all reps are hit. This is the core of a proper strength program for a swimmer.

Rotation-Aware Exercise Selection

pickFresh() deprioritising exercises used in the last 2 sessions prevents repetitive boredom and overuse patterns. Small detail, meaningful for long-term adherence.

Tempo Notation

Every weighted exercise has a tempo code (e.g. 3-1-1-0) with an interactive explainer modal. Most general fitness apps don't touch this. For a swimmer whose strength purpose is very specific — start power, kick force, pull strength — tempo is non-negotiable and it's here.

Goal Program Architecture

The separation of long-term aspirational goals from coach-assigned training focus is conceptually correct and rare. The TRAINING_FOCUS map driving exercise bias is practical and extensible. The meet-to-phase timeline visualisation is clean.

Schedule & Taper Logic

Season template + per-day overrides + automatic dryland taper when inside a meet window. The schedule is the source of truth for workout type — not just a calendar. Priority-A meets trigger a heavier taper than B-meets. These are the right design decisions.

Active Bugs

Fix First
🔴 Unit Picker Visual Toggle Not Reflecting in App
The kg/lbs and km/mi buttons in the More tab visually update when tapped (inline style is being applied correctly via stylePicker()), but the unit change doesn't reflect in other tabs without switching away and back. Root cause: applyUnitSettings() calls renderKmTracker(), renderHomeCals(), and renderBaseline() — but none of these update the currently visible tab if the user is sitting on More. More critically, there is still a bug in renderKmTracker(): line 1038 reads useKm (no S. prefix) which will fail silently. Fix: change useKm to S.useKm on that line. After that, the unit label in the km tracker will update correctly when the More tab is active.
🔴 km Tracker Uses Wrong Variable Reference
Line ~1038: const lastDisp=useKm?last.km... — useKm is not in scope. Should be S.useKm. This silently breaks the last-entry label display in imperial mode and is likely contributing to the perceived unit toggle failure.
🟠 Onboarding Units Dropdown Sets useKg But Not useKm
In saveProfile() during onboarding, S.useKg is set from the metric/imperial preference but S.useKm is never set — it stays at the default true regardless of what the athlete chose. A swimmer who selects Imperial at setup will get lbs in the gym but km in the distance tracker, which is inconsistent. Fix: add S.useKm = unitsPref !== 'imperial'; DB.set('useKm', S.useKm); alongside the existing useKg line.
🟡 Timer Notifications Require One-Time Permission Setup
Not exactly a bug, but the notification permission request is silent on first run — there's no UI prompt explaining why the app is asking or confirming that it's a one-time ask. Swimmers get a random browser permission dialog with no context. Add a short inline explanation in the timer card before the first permission request fires.

Critical Gaps

Missing
1. Fuel Tab is a Calculator, Not a Tool

The calorie estimate is correct and the macro split (2.4g protein/kg for a swimmer) is appropriate. But that's where it stops. There is no food log, no meal timing framework, no swim-specific guidance on pre/intra/post-training nutrition. A swimmer at this level is eating 4,000–5,000 calories/day across 5–7 eating occasions. Knowing the target number is step one. Step two — what to actually eat and when — is completely absent.

What's needed (minimum viable): Pre-training meal window (2–3h before, carb-heavy), intra-training guidance (if double sessions, 30–60g fast carbs between), post-training window (30–45min, protein + carb combined). These are not complex — they're fixed rules for competitive swimmers that can be displayed as a daily checklist or timeline.

What's needed (next level): A simple food log — not a full MyFitnessPal clone, but logging meals as pre/intra/post categories with macro ballpark estimates. Even 5 food entries per occasion would give meaningful feedback.

2. Times Tab Has No Trajectory or Goal-Date Logic

The LCM Analysis card calculates improvement rate and projects months to standard. That's good. But there's no connection between that projection and the actual meet calendar. The app knows when the target meet is. It knows the current gap to standard. It knows the historical improvement rate. It should be saying: "At your current rate of 0.12s/month, you'll hit the AB Provincials standard with 3 weeks to spare. You need to find 0.05s more per month to also qualify for Trials." That's actionable. Right now the numbers are shown but the conclusion is never drawn.

What's also missing: No split-time entry. Competitive swimmers track 50m splits, not just final times. A 100 Free swimmer cares deeply whether their back-half split is dying. No relay splits. No splits from practice (timed swims). This is a complete picture problem — the data model only stores a final time per meet.

3. No Pool-Side Session Log

The schedule tab shows what the swimmer is supposed to do in the pool. There's no place to record what actually happened — yardage, sets, efforts, how a timed swim went, how they felt in the water. This means the app has no feedback loop between pool performance and the dryland program. A swimmer who has a terrible practice doesn't have a way to log that and have it influence tomorrow's dryland load (beyond the readiness slider, which is a blunt instrument).

This doesn't need to be a full swim log. Even a post-practice check-in: "How was the pool session? / Timed effort today? / Yardage approx." — that data is useful and it's completely absent.

4. No Hydration Tracking in the Fuel Tab

The daily water target is calculated correctly (weight × 40ml + 550ml per session). It's shown in the Fuel tab. But there's no way to log actual intake. Dehydration at 2% of bodyweight drops swim performance by 1–2% — for a swimmer trying to drop 0.3s, that's meaningful. This is a simple tally — "cups logged today" would be sufficient.

5. No Stretch / Mobility Progress Tracking

The mobility exercises are excellent and correctly sequenced (shoulder sleeper stretch, lat stretch, ankle mobility, hip flexors — all directly tied to swimming mechanics). But there's no way to track progression on mobility. Did the hip flexor lunge improve? Is the ankle mobility drill getting easier? Without logging, it's just a checklist. Mobility is where many swimmers drop the ball, and accountability matters. Even a simple "sessions completed this week" counter would help.

6. No Athlete Body Composition Trend

The app has a bodyStats object and a lastStatsUpdate timestamp. But there's no visible weight/body composition trend chart in the app. If a swimmer is logging their weight (implied by the presence of bodyStats), they should be able to see the trend over time — particularly in the context of race weight vs. training weight. This data exists but isn't surfaced.

7. No RPE (Rate of Perceived Exertion) Logging on Workouts

Each exercise has a set/rep/weight log. But there's no post-session RPE or session load score. In elite training, session RPE (session exertion × duration = training load in arbitrary units) is the most practical load monitoring tool. Without it, the only load data is readiness going in — there's no measure of how hard the session actually was. One number (1–10 RPE at session end, automatically multiplied by approximate session duration) would close this loop and allow proper TSS-equivalent tracking over time.

8. No Baseline Strength Testing Protocol

The 1RM system is solid, but there's no structured baseline test built into the app. A swimmer starting fresh has no 1RM data, so the percentage loading system sits idle until they manually enter estimates. A guided "Set Your Baselines" flow — built as part of onboarding or as a first-week protocol — would unlock the loading system immediately and give the data needed to make the program work from day one.

Redundant or Weak Sections

Tighten
The "Preview Any Workout" Button Is Redundant in Goal Mode

In goal program active state, the "Preview any workout type" button appears alongside "Generate Goal Workout." If the athlete is in goal mode, the preview modal forces a workout type that bypasses all the goal biasing — so they get a standard workout. This is confusing. The preview button should either be removed from goal mode entirely, or relabelled "Override today's type" with a clear warning that it bypasses goal biasing.

Season km Tracker Is Disconnected from Everything

The km tracker accumulates total pool distance manually entered by the athlete. But this number doesn't connect to calorie calculations (which use session count, not distance), isn't tied to standards progression, and doesn't appear in the Times tab analysis. It's floating data. Either integrate it — feed it into the nutrition calculation, show it against a season target, or tie it to a milestone progression — or remove it. Right now it's a number that exists without meaning.

Program Week / Cycle Counter Is Semi-Hidden and Confusing

The 16-week cycle counter auto-advances every 3 logged workouts. This is a simple approximation and it works — but the athlete can't see the logic, can't manually adjust it, and can't reset it if their season doesn't align to 16-week blocks. A small "Week X of 16 — adjust" control in the More tab would make the system feel intentional rather than opaque.

Long-Term Goals Are Set But Do Nothing

The LONG_TERM_GOALS array (Make the Olympic team, Earn a US scholarship, etc.) is displayed in the profile and selected during onboarding. But per the design decision made during build, they don't actually influence any part of the workout or standards system. That's fine as a temporary state — but a user who picks "Earn a US scholarship" and then sees a generic workout with no reference to their goal will wonder why they were asked. Either wire it up minimally (show a motivational note in the workout header based on their goal) or remove it from the profile display so it doesn't look like a broken feature.

Fuel Tab Height Entry Inconsistency

In metric mode: height is entered as cm. In imperial mode: as ft + inches. The conversion math is in ftInToCm() and it works, but the UI rebuilds the height row on unit switch (handled in renderNutrition()), and applyUnitSettings() has had this function removed from it, meaning a unit switch while the Fuel tab is open doesn't update the height field layout. Minor but worth noting as part of the unit toggle cleanup.

App Flow Assessment

UX
✅ What flows well
✓Onboarding is clean and purpose-driven — name, events, standard, goals. Gets the athlete to the app immediately.
✓Home tab is a proper daily dashboard — greeting, phase info, schedule summary, next meet countdown, readiness check-in all visible without scrolling far.
✓Workout generation is one tap. The phase badge, load alert, and schedule source note give context without being verbose.
✓Times tab is the strongest content tab — event selector, PB display, gap pills, progress bar, chart, analysis card. Good structure.
✓Modal architecture (exercise info, tempo, schedule editor) keeps the main flow clean.
⚠️ What doesn't flow well
⚠The Fuel tab is the weakest page in the app. You fill in body stats, see numbers, and then... nothing to do. No daily actions, no logging, no feedback loop. It's a dead end.
⚠The More tab is a dumping ground — settings, profile, backup, logo upload, danger zone, season setup all live there. It needs grouping. The settings card (units) is buried at the bottom after all the profile and logo content.
⚠The Schedule tab requires understanding of the season template concept before it makes sense. A new user lands there and sees an empty calendar with no affordance for what to do first.
⚠Goal program → standard mode toggle uses the same tab, which means two distinct modes are collapsed into one Workout section with a small toggle at the top. A first-time user might not notice the toggle exists.
⚠No success/completion state after finishing a workout beyond a confirm dialog and route to Home. A brief summary of what was completed (sets, estimated load, phase note) would feel like a reward.
